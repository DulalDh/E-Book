CHAPTER 5

System Design

5.1  Introduction

In this chapter, discus about system design and the system design is divided into three part such as architectural design, logical design and physical design. Systems design is a systematic approach to the design of a system. It may take a bottom-up or top-down approach, but either way the process is systematic wherein it takes into account all related variables of the system that needs to be created—from the architecture, to the required hardware and software, right down to the data and how it travels and transforms throughout its travel through the system. Systems design then overlaps with systems analysis, systems engineering and systems architecture. 

 

5.2 Architectural Design

The architectural design of a system emphasizes the design of the system architecture that describes the structure, behavior and more views of that system and analysis. Requirements of the software should be transformed into an architecture that describes the software's top-level structure and identifies its components. This is accomplished through architectural design (also called system design), which acts as a preliminary 'blueprint' from which software can be developed.

 

Class diagram:

Class diagram describes the attributes and operations of a class and also the constraints imposed on the system. The class diagrams are widely used in the modeling of object oriented systems because they are the only UML diagrams, which can be mapped directly with object-oriented languages.

Figure 5.2 Class diagram of SASTC information

 

5.3 Logical design

The logical design of a system pertains to an abstract representation of the data flows, inputs and outputs of the system. This is often conducted via modelling, using an over-abstract (and sometimes graphical) model of the actual system. In the context of systems, designs are included. Logical design includes entity-relationship diagrams logical design is critical to the implementation of a corporate database. If your logical design is incomplete or has flaws, making changes to the means of data collection, storage, and protection can be costly later on. By using a well-conceived preliminary design, you can easily implement and test a database. A sound logical design therefore helps to ensure a successful implementation.

ER Diagram of Bus location tracking system:

Fig 5.1: ER Diagram of Bus location tracking system

5.4 Physical design

The physical design relates to the actual input and output processes of the system. This is explained in terms of how data is input into a system, how it is verified/authenticated, how it is processed, and how it is displayed. In physical design, the following requirements about the system are decided. During the physical design process, you convert the data gathered during the logical design phase into a description of the physical database, including tables and constraints. Physical design decisions, such as the type of index or partitioning have a large impact on query performance.

Data follow Diagram:

A Data Flow Diagram (DFD) is traditional visual representation of the information flows within a system. A neat and clear DFD can depict a good amount of the system requirements graphically. It can be manual, automated, or combination of both.

It shows how information enters and leaves the system, what changes the information and where information is stored. The purpose of a DFD is to show the scope and boundaries of a system as a whole. It may be used as a communications tool between a systems analyst and any person who plays a part in the system that acts as the starting point for redesigning a system.

It is usually beginning with a context diagram as the level 0 of DFD diagram, a simple representation of the whole system. To elaborate further from that, we drill down to a level 1 diagram with lower level functions decomposed from the major functions of the system. This could continue to evolve to become a level 2 diagram when further analysis is required. Three (zero, first & second) level DFD show in following figure:

 

Fig 5.2 : Level 0 Data Flow Diagram

Fig 5.2 : Level 1 Data Flow Diagram

 

Fig 5.2 : Level 1 Data Flow Diagram

5.5 Choosing Implementation Model

Implementation models are used in each system. There are many models in the present world and they have a lot of different steps that can be used to make the system properly. That's why we are also used waterfall model for our system that show in following

 

Waterfall Model

The waterfall model is a sequential design process, used in software development  processes, in

which progress is seen as flowing steadily downwards  (like a waterfall) through  the phases of

Conception, Initiation, Analysis, Design, Construction, Testing , Production/Implementation  and

Maintenance. The waterfall development  model originates in the manufacturing and construction  industries:  highly structured physical environments  in which after-the-fact changes are prohibitively costly,  if not impossible. Since no formal software development  methodologies existed at the time, this hardware-oriented  model was simply adapted for software development

 

Diagram:

Fig 5.4: Waterfall model

 

Why use the waterfall model

This model is simple and easy to understand and use. In, this model phases are processed and completed  one at a time and phases do not overlap. Waterfall model works well for smaller projects where sequence are very well understood.

 

 

5.6 Conclusion

In this chapter process model was introduced and used to design a high level framework on top of the Java Database Connectivity (JDBC) is an application programming interface (API).  It was decided during analysis that a database application would also be developed to test the framework.  This will enable the framework to be evaluated and possibly determine any additional functionality required.
